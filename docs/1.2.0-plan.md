# WICompress 1.2.0 落地计划

本文档把 roadmap 里的 target bytes 讨论收敛成 v1.2.0 可实现计划。目标是在不改变现有策略式 API 行为的前提下，新增一套结果约束式压缩模式。

## 发布目标

- 新增目标式压缩入口，支持分享/上传约束，例如“最长边不超过 512 px，编码后数据不超过 2 MB”。
- 保持现有 `WICompressOptions` API 的源码兼容和行为兼容。
- 保持输出约束显式可配置，例如格式和 metadata。
- 色彩空间策略只有在实现和测试都足够直接时才进入 v1.2.0，避免把本次发布扩大成完整输出模型重构。
- 继续保持 UIKit/AppKit-free 的 ImageIO core。

## 非目标

- 不把 `maxBytes` 加进 `WICompressOptions`。
- 不暴露内部搜索参数，例如最大尝试次数、最低质量、最小长边、quality knee 阈值。
- 不为了满足目标大小而静默改变输出格式。PNG 仍然输出 PNG，除非调用方显式请求其他格式。
- v1.2.0 不做 PNG 调色板量化。
- 不改变现有 API 的公开返回类型。
- 本次不引入 Photos/PhotosUI adapter。

## Public API 形态

现有策略式 API 保持为专家/过程控制模式：

```swift
public static func compress(
    _ data: Data,
    options: WICompressOptions = .default
) throws(WICompressError) -> Data
```

v1.2.0 新增目标/结果约束模式：

```swift
public struct WICompressionTarget: Sendable, Equatable {
    public var maxBytes: Int
    public var maxLongSide: Int?
    public var format: WIFormatPolicy
    public var metadata: WIMetadataPolicy

    public init(
        maxBytes: Int,
        maxLongSide: Int? = nil,
        format: WIFormatPolicy = .preserve,
        metadata: WIMetadataPolicy = .strip
    )
}

extension WICompress {
    public static func compress(
        _ data: Data,
        to target: WICompressionTarget
    ) throws(WICompressError) -> Data
}
```

命名说明：

- `maxBytes` 表示硬性字节上限，不表示要尽量填满字节预算。
- `maxLongSide` 表示最终 EXIF-oriented display long side 不能超过该值。它和现有 `.maxPixel(Int)` 概念对应，但在 target API 边界更明确。
- `format` 和 `metadata` 是输出约束，不是搜索参数。

参数校验：

- `maxBytes <= 0` 非法。
- `maxLongSide <= 0` 非法。
- `WICompressionTarget` initializer 不抛错；非法 target 在
  `compress(_:to:)` 编码开始前抛错。

新增错误：

```swift
public enum WICompressError {
    case invalidCompressionTarget
    case targetBytesUnreachable(maxBytes: Int, bestByteCount: Int?)
}
```

`targetBytesUnreachable` 表示编码器无法在满足输出约束的同时生成 `count <= maxBytes` 的数据。

## 色彩空间范围

色彩空间策略有价值，但 v1.2.0 的范围应保持小。如果纳入本次发布，只做最小 API：

```swift
public enum WIColorSpacePolicy: Sendable, Equatable {
    case preserve
    case sRGB
    case displayP3
}
```

集成选择：

- 只有当实现足够直接，并且测试能够覆盖 sRGB、Display P3 和至少一条转换路径时，才考虑同时给 `WICompressOptions` 和 `WICompressionTarget` 增加 `colorSpace`。
- 否则推迟到 v1.3.0，让 v1.2.0 聚焦 target compression。

色彩空间规则：

- Color profile 是显示语义，不是隐私 metadata。
- Metadata stripping 不应移除输出 color profile。
- 当显式色彩空间转换存在时，JPEG alpha flattening 必须在解析后的输出色彩空间中完成合成。

## 内部架构

新增独立 target solver，不把 `WIWritePlanResolver` 扩展成迭代引擎：

```text
Data
  -> WIImageSource
  -> WICompressionTarget validation
  -> WITargetCompressionSolver
       -> candidate WIWritePlan values
       -> WIImageEncoder.encode(...)
       -> candidate ranking
  -> final hard byte check
  -> Data
```

职责划分：

- `WIWritePlanResolver`：继续负责一次性策略压缩的 plan 解析。
- `WIImageEncoder`：继续负责执行一个具体 plan。
- `WITargetCompressionSolver`：负责 target search、candidate 收集和最终选择。
- `WICompressionTarget`：只承载公开 target 约束。

预期内部模型：

```swift
struct WITargetCandidate {
    var data: Data
    var byteCount: Int
    var longSide: Int
    var quality: Double?
    var hitQualityKnee: Bool
}
```

Solver 返回前必须始终执行最终硬检查：`data.count <= target.maxBytes`。

## 目标压缩算法

Target mode 是一个约束搜索过程。它在内部同时控制 resolution 和 lossy quality。

### 共享准备阶段

1. 用 `WIImageSource` 检查 source。
2. 根据 `target.format` 解析 destination format。
3. 尽量复用现有错误，拒绝不支持的 source 或 destination format。
4. 如果透明 source 转 JPEG，但 `WIFormatPolicy.jpeg(background:)` 没有提供 flatten background，则拒绝。
5. 计算 EXIF-oriented display dimensions。
6. 计算 reference long side：

```text
referenceLongSide = min(sourceDisplayLongSide, target.maxLongSide)
referenceLongSide = sourceDisplayLongSide when target.maxLongSide is nil
```

7. 只有同时满足以下条件时才允许直接返回原始数据：

- `data.count <= target.maxBytes`
- format policy 是 `.preserve`
- metadata policy 已经满足
- orientation 已经和 metadata policy 兼容
- display long side 已经满足 target
- 如果未来存在 color-space policy，也必须已经满足

### JPEG 和 HEIC

使用尺寸作为外层搜索，固定尺寸下的 quality search 作为内层搜索。

内部 quality profile，不进入 public API：

```text
qHigh = 0.82       // 快速成功质量
qAnchor = 0.72     // 缩小尺寸后重新尝试的质量
qKnee = 0.45       // 低于该值后优先缩尺寸
qEmergency = 0.25  // 只用于极端兜底
```

算法：

1. 从原始 source render 到当前 long side。
2. 用 `qHigh` 编码。
3. 如果结果满足 `maxBytes`，记录候选；当当前尺寸就是 reference size 时可立即返回。
4. 用 `qKnee` 编码。
5. 如果 `qKnee` 满足 `maxBytes`，在 `[qKnee, qHigh]` 中用小步数边界搜索找到最高可行 quality。
6. 如果最高可行 quality 接近 `qKnee`，额外 probe 一个更小尺寸 + `qAnchor`，避免返回高分辨率、极低质量的结果。
7. 如果 `qKnee` 仍然超预算，用面积比例估算下一轮更小 long side：

```text
nextLongSide = currentLongSide * sqrt(maxBytes / encodedBytesAtKnee) * safety
safety = 0.88
```

8. Clamp 下一轮 long side，保证它小于当前值、至少为 `1`，并在 destination encoder 受益时对齐到偶数尺寸。
9. 重复搜索，直到找到候选或耗尽内部 encode budget。
10. 如果仍没有候选，尝试更小尺寸 + `qEmergency` 的 emergency path。
11. 仍失败则抛出 `targetBytesUnreachable`。

Quality search 细节：

- 把 quality/byte 关系视为“大体单调”，但不能依赖严格单调。
- 缓存已经编码过的 quality，避免重复工作。
- 保存每个成功候选。
- 不能因为尝试次数耗尽就返回超限结果。
- 如果已经存在成功候选，且预算耗尽，则返回当前最好的成功候选。

v1.2.0 candidate ranking：

1. 优先选择没有低于 `qKnee` 的候选。
2. 优先选择更大的 long side。
3. 优先选择更高 quality。
4. byte count 只作为最后的平局因素。

### PNG

ImageIO 下的 PNG 没有常规 lossy quality knob。v1.2.0 只使用尺寸搜索。

算法：

1. Render 到 reference long side。
2. 编码一次 PNG。
3. 如果满足 `maxBytes`，直接返回。
4. 如果超预算，用同样的面积比例估算下一轮 long side：

```text
nextLongSide = currentLongSide * sqrt(maxBytes / encodedBytes) * safety
```

5. 重复直到找到候选或图像到达 `1x1`。
6. 如果没有候选，抛出 `targetBytesUnreachable`。

规则：

- 不自动把 PNG 改成 JPEG。
- 不暴露或伪造 PNG quality。
- v1.2.0 不做 palette quantization。

## 实现步骤

1. 新增 `WICompressionTarget`。
2. 新增 target 相关 `WICompressError` case。
3. 新增 `WICompress.compress(_:to:)` 和 URL overload：

```swift
public static func compress(
    contentsOf url: URL,
    to target: WICompressionTarget
) throws(WICompressError) -> Data
```

4. 新增 `WITargetCompressionSolver`。
5. 新增少量内部 helper：

- oriented display dimensions
- 使用 override `maxPixelSize` 创建 plan
- 使用 override `quality` 创建 plan
- next-long-side prediction
- candidate ranking

6. 尽量保持 `WIImageEncoder` 不变。只有当现有 `WIWritePlan` 无法表达 target candidate 时，才增加 overload 或字段。
7. 先补测试，再更新大段文档。
8. 验证行为后更新 README 和 README_CN。
9. 更新 CHANGELOG 的 v1.2.0 条目。

## 测试计划

Public API 测试：

- `compress(_:to:)` 拒绝 `maxBytes <= 0`。
- `compress(_:to:)` 拒绝 `maxLongSide <= 0`。
- `compress(_:to:)` 返回的 data count `<= maxBytes`。
- URL target overload 将读取失败映射为 `fileReadFailed`。

JPEG/HEIC target 测试：

- JPEG target with `maxLongSide: 512` and `maxBytes: 2 * 1024 * 1024` 可以快速成功，并同时满足两个限制。
- 小 byte limit 的 JPEG target 会触发 quality search。
- 无法满足的 JPEG target 抛出 `targetBytesUnreachable`。
- 当前平台可写 HEIC 时，HEIC target 保持 HEIC 输出。
- 显式 JPEG conversion 对透明 PNG 且没有 background 的场景继续拒绝。
- 显式 JPEG conversion 搭配 `.white` 或 `.black` background 可以成功。

PNG target 测试：

- PNG target 满足 `maxLongSide`。
- PNG target 通过缩尺寸满足 `maxBytes`。
- PNG target 不转换为 JPEG。
- 不可能满足的 PNG target 抛错。
- PNG alpha 在 target compression 后仍保留。

Policy correctness 测试：

- `.metadata(.strip)` 不能通过 target passthrough 返回带 metadata 的原图。
- `.metadata(.preserve)` 只有在原图已经满足 byte 和 dimension limit 时才允许返回原图。
- strip metadata 时不能绕过 orientation normalization。
- 显式 destination format 不能通过 passthrough 返回 source-format bytes。

Regression 测试：

- 现有 `swift test` 全部继续通过。
- 现有 strategy API 默认值保持 `.luban`、`.preserve`、`.strip`、`.compression(0.6)`。

## Fixture 需求

现有 fixtures 已覆盖大部分 v1.2.0 工作：

- JPEG landscape
- 用于 size-guard 行为的 recompressed JPEG
- PNG alpha image
- PNG panoramas
- HEIC orientation/GPS/HDR fixtures
- animated GIF rejection

可能需要新增的 fixtures：

- 一个需要降低尺寸才能满足真实 byte target 的大 PNG。
- 一个极小 target fixture，用于证明 `targetBytesUnreachable`。
- 如果 color-space policy 进入本次发布，除现有 Display P3 fixtures 外，还需要增加 sRGB fixture，并至少覆盖一个 conversion assertion。

## 文档更新

README 和 README_CN 需要解释两种模式：

```swift
// Strategy mode: caller controls resize and quality.
let data = try WICompress.compress(
    input,
    options: WICompressOptions(
        resize: .maxPixel(1600),
        format: .preserve,
        metadata: .strip,
        quality: .compression(0.7)
    )
)

// Target mode: caller declares final output constraints.
let thumbnail = try WICompress.compress(
    input,
    to: WICompressionTarget(
        maxBytes: 2 * 1024 * 1024,
        maxLongSide: 512,
        format: .jpeg(background: .white),
        metadata: .strip
    )
)
```

文档必须说明：

- Target mode 拥有内部 quality 和 resolution 决策权。
- `maxLongSide` 是上限；solver 为满足 `maxBytes` 可以输出更小的图。
- `maxBytes` 是硬限制；如果没有合法候选能满足它，compression 会抛错。
- PNG target compression 不会静默转换格式。
